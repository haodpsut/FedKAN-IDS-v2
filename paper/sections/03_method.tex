\section{The FedKAN-IDS Framework}\label{sec:method}

This section describes how Definitions~\ref{def:kan} and~\ref{def:fl-ids} are instantiated in the FedKAN-IDS pipeline. Section~\ref{subsec:arch} fixes the per-client model architecture and the global system flow (Fig.~\ref{fig:flow}); Section~\ref{subsec:server} formalises the server orchestration (Algorithm~\ref{alg:server}); Section~\ref{subsec:client} details the client local optimisation (Algorithm~\ref{alg:client}).

Each IoT client $C_k$ instantiates a single-hidden-layer KAN
\begin{equation}
f(\Theta;\mathbf{x}) \;=\; \Phi^{(\text{out})}\big(\Phi^{(\text{hid})}(\mathbf{x})\big),
\end{equation}
where $\Phi^{(\text{hid})}: \mathbb{R}^d \to \mathbb{R}^{n_h}$ and $\Phi^{(\text{out})}: \mathbb{R}^{n_h} \to \mathbb{R}^{c}$ are KAN layers (Definition~\ref{def:kan}) with shared grid size $G$ and order $k$. The model output is passed through a softmax to produce class probabilities; we train with cross-entropy loss. The client parameter set $\Theta = \{w^{(b)}, w^{(s)}, c\}$ comprises three structurally distinct sub-tensors which the server aggregates element-wise.

For NetFlow-v2 IDS we fix $d = 39$ (the $43$ original features minus the four identifier columns dropped during preprocessing, see Section~\ref{sec:experiments}), and $c \in \{2, 5\}$ for binary and 5-class detection on NF-BoT-IoT-v2. Throughout, our default architecture is $n_h = 8$, $G = 5$, $k = 3$, giving $|\Theta| \approx 3{,}300$ parameters, well within the TinyML envelope of microcontroller-class IoT gateways.

\subsection{Server Orchestration}\label{subsec:server}

The server runs a synchronous Federated Averaging loop adapted to the three KAN parameter sets (Algorithm~\ref{alg:server}). Each round, a fraction $C \in (0, 1]$ of the $K$ clients is selected; in our experiments $C = 1$ (full participation), but the algorithm and analysis remain valid for $C < 1$. The aggregation step (line~\ref{alg-server:agg}) performs element-wise weighted averaging on each of $\{w^{(b)}, w^{(s)}, c\}$ in parallel. Because the B-spline coefficients $c$ enter the edge function~\eqref{eq:edge} linearly, FedAvg over $c$ is mathematically equivalent to averaging the underlying spline approximations under a fixed grid; this is the property exploited in Lemma~\ref{lem:spline} and Theorem~\ref{thm:conv}.

\begin{algorithm}[t]
\caption{FedKAN-IDS Server Orchestration}
\label{alg:server}
\begin{algorithmic}[1]
\Require Number of clients $K$, sampling fraction $C$, rounds $T$, KAN hyperparameters $(n_h, G, k)$.
\Ensure Trained global parameters $\Theta_T$.
\State Initialise $\Theta_0 = \{w^{(b)}_0,\, w^{(s)}_0,\, c_0\}$
\For{$t = 0, 1, \ldots, T-1$}
    \State $m \leftarrow \max(\lceil C \cdot K \rceil,\, 1)$
    \State $\mathcal{S}_t \leftarrow$ random subset of $m$ clients from $\{1,\ldots,K\}$
    \For{each client $k \in \mathcal{S}_t$ \textbf{in parallel}}
        \State $\Theta^{k}_{t+1} \leftarrow \Call{ClientUpdate}{k, \Theta_t}$
    \EndFor
    \State $\Theta_{t+1} \leftarrow \displaystyle\sum_{k \in \mathcal{S}_t}
                  \frac{N_k}{\sum_{j \in \mathcal{S}_t} N_j}\,\Theta^{k}_{t+1}$
                  \label{alg-server:agg}\Comment{element-wise on each of $w^{(b)}, w^{(s)}, c$}
\EndFor
\State \Return $\Theta_T$
\end{algorithmic}
\end{algorithm}

\subsection{Client Local Optimisation}\label{subsec:client}

\begin{algorithm}[t]
\caption{FedKAN-IDS \textsc{ClientUpdate}}
\label{alg:client}
\begin{algorithmic}[1]
\Require Client index $k$, broadcasted parameters $\Theta_t$, local data $\mathcal{D}_k$, local epochs $E$, batch size $B$, learning rate $\eta$.
\Ensure Updated local parameters $\Theta^{k}$.
\State $\Theta^{k} \leftarrow \Theta_t$
\State partition $\mathcal{D}_k$ into batches of size $B$
\For{epoch $e = 1, \ldots, E$}
    \For{minibatch $(\mathbf{x}, \mathbf{y}) \in \mathcal{D}_k$}
        \State \textbf{Forward:} for each KAN edge $(q, p)$,
        \[
            \phi_{q,p}(x_p) = w^{(b)}_{q,p}\,\sigma(x_p) + w^{(s)}_{q,p} \sum_{i=1}^{G+k} c_{q,p,i}\, B_i(x_p)
        \]
        \State $\hat{\mathbf{y}} = f(\Theta^k; \mathbf{x})$ \Comment{stack the two KAN layers}
        \State $\mathcal{L} = \mathrm{CrossEntropy}(\hat{\mathbf{y}}, \mathbf{y})$
        \State \textbf{Backward:} compute $\nabla_{\Theta^k}\mathcal{L}$
        \State \textbf{Update:} $\Theta^{k} \leftarrow \Theta^{k} - \eta\, \nabla_{\Theta^k}\mathcal{L}$
        \Statex \quad \Comment{Adam in our experiments; SGD also valid}
    \EndFor
\EndFor
\State \Return $\Theta^{k}$ to server
\end{algorithmic}
\end{algorithm}

Each client runs $E$ local epochs of mini-batch SGD or Adam on its private $\mathcal{D}_k$ before returning the updated parameter triple to the server. Backpropagation through a KAN layer is straightforward: $\partial \phi_{q,p}/\partial w^{(b)}_{q,p} = \sigma(x_p)$, $\partial \phi_{q,p}/\partial w^{(s)}_{q,p} = \sum_i c_{q,p,i} B_i(x_p)$, and $\partial \phi_{q,p}/\partial c_{q,p,i} = w^{(s)}_{q,p} B_i(x_p)$, all of which are computed automatically by modern autodiff frameworks.

In our implementation we use the open-source \texttt{efficient\_kan} library, which fuses the spline-coefficient lookup into a single contiguous matrix multiplication and reduces the constant factor in the per-edge forward cost relative to a naive double-summation implementation. The full pipeline (data partitioning, FedAvg server, KAN client) is released as the open-source repository accompanying this work; configurations and run-level metrics for all 146 experimental runs reported in Section~\ref{sec:experiments} are committed to that repository.

\subsection{Computational and Communication Footprint}\label{subsec:cost}

For the configurations used in Section~\ref{sec:experiments}, FedKAN-8 transmits $|\Theta_{\text{KAN}}| \cdot 4$ bytes $\approx 13.1$~kB per client per round in float32, and the parameter-matched MLP-PM-80 transmits $\approx 13.4$~kB per client per round. Per-round wall-clock on a single NVIDIA A100 (excluding the first-round CUDA warm-up) is $2.1$--$2.8$~s for FedKAN and $1.8$--$2.4$~s for MLP-PM, both well under the per-round budget of any realistic IoT FL deployment. We discuss FLOPs and peak-memory profiling, including measurements on Raspberry Pi-class hardware, in Section~\ref{sec:experiments}.
